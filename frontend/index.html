    what we check

    Four real signals, not a guess

    Each scan runs the same checks a security-conscious person would do by hand — just automated and queued as a background job.

    ◈

    DNS resolution

    Resolves the domain's A and MX records. A domain that doesn't resolve at all, or resolves to nothing meaningful, is treated as a risk signal on its own.

    ◷

    WHOIS age

    Looks up when the domain was registered. Phishing domains are frequently registered days or hours before use — age under 30 days raises the score.

    ⛨

    TLS certificate

    Checks whether HTTPS is actually configured and the certificate hasn't expired. Missing or expired certs are a meaningful red flag on their own.

    ◎

    Reputation signals

    Cross-references a blacklist, checks for phishing-style keywords in the URL (login, verify, secure...), and optionally Google Safe Browsing's threat list.

    how it works

    An actual job queue, not a spinner

    Submitting a URL doesn't run the checks inline — it queues real background work.

    1

    You submit a link

    The API validates it, checks the cache for a recent result on that domain, and returns instantly if found.

    2

    A job is queued

    Otherwise a scan job is created and handed to a Celery worker — the request doesn't block waiting for it.

    3

    Checks run async

    DNS, WHOIS, TLS, and reputation checks run in the background while the page polls for status.

    4

    Verdict + caching

    A risk score and verdict are computed and stored, then cached by domain so the next scan of it is instant.

    /// faq

    Common questions

    Does this guarantee a link is safe?

    No — it's a risk signal, not a guarantee. A low score means the checks found nothing suspicious; it doesn't replace judgment, especially for links asking for credentials or payment.

    Why does the first scan of a domain feel slower?

    The first scan runs the full pipeline (DNS, WHOIS, TLS, reputation) as a background job. Repeat scans of the same domain are served from cache and return almost instantly.

    Is my scan history private?

    History is stored so you (and this demo) can browse past scans and export them as JSON. It isn't tied to a login — anyone with access to this instance can see the shared history.

    What happens with a suspicious or malicious verdict?

    You'll see which signals triggered it, and an option to report the link instead of opening it. Opening it anyway is still possible — the tool informs, it doesn't block.
